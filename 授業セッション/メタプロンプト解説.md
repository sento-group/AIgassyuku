# 領域適応型ドキュメント生成プロンプト 解説資料

## このプロンプトが解決する課題

### 課題1：専門領域ごとに必要な資料がバラバラで、何を作ればいいかわからない

**例えば...**

- 営業なら「トークスクリプト」「QA集」が必要
- マーケティングなら「市場分析」「SWOT」が必要
- でも、毎回「何が必要か」を考えるのは面倒

**解決策**
→ **領域を自動判定して、その領域の標準カテゴリを提案**

- ユーザーが「新規事業やりたい」と言えば、AIが「市場分析・競合分析・財務モデル...」と提案
- **ユーザーは「何を作るべきか」を考える手間がゼロになる**

---

### 課題2：1カテゴリずつ対話すると時間がかかりすぎる

**例えば...**

- 「まず市場分析について教えてください」
- 「次に競合分析について...」
- 「次にビジネスモデルについて...」
- → **10カテゴリあったら10回往復が必要で、めちゃくちゃ時間がかかる**

**解決策**
→ **全カテゴリのたたき台を一気に出力**

- ユーザーが気になるところだけ修正すればいい
- **効率が10倍になる**

---

### 課題3：AIが勝手に推測した内容が的外れなことがある

**例えば...**

- 「あなたの会社のミッションは●●ですね」→ 全然違う！
- 「価格はこれくらいがいいでしょう」→ それは私が決めたい！
- → **ユーザー固有の情報をAIが勝手に埋めると、信頼が下がる**

**解決策**
→ **情報源を3タイプに分類して、自動判断**

| 情報タイプ | AI の行動 | 例 |
| --- | --- | --- |
| **客観的事実** | Web検索して埋める | 市場規模、競合の資金調達額 |
| **推論可能な一般論** | 仮説として提示（要確認フラグ付き） | 一般的なSaaSのKPI |
| **ユーザー固有情報** | プレースホルダー化（勝手に埋めない） | 自社のMVV、価格戦略 |

**結果：ユーザーは「これは調べてくれた情報」「これは自分で決める情報」が明確にわかる**

---

### 課題4：AIが勝手にリサーチを始めると、何をやってるかわからず不安

**例えば...**

- いきなり「リサーチ中...」と出る
- 「なんでこれ調べてるの？」「本当に必要？」と不安になる
- → **ブラックボックス化すると、ユーザーの納得感が下がる**

**解決策**
→ **リサーチ判断の理由を説明して、許可を取る**

```
【リサーチ提案】
以下のカテゴリはWeb検索でデータ取得が有効です：

- カテゴリ1「市場機会分析」
  理由：市場規模・成長率などの客観データが公開されているため

- カテゴリ2「競合分析」
  理由：競合企業の公開情報（価格・資金調達等）が取得可能なため

これらのリサーチを実施してもいいですか？

```

**結果：ユーザーは「なぜリサーチするのか」を理解した上で許可できる → 納得感が高まる**

---

### 課題5：領域判定が間違ってると、全部ズレた提案になる

**例えば...**

- ユーザー：「新規事業やりたい」
- AI：（勝手に）「これは営業の話ですね！トークスクリプト作ります」
- → **最初の判定ミスで、全てが無駄になる**

**解決策**
→ **推論結果を必ずユーザーに確認**

```
【領域推論】
あなたの相談は「新規事業（SaaS領域）」と理解しました。
背景として「B2B SaaS、アーリーステージ、事業計画とGTM準備」を想定していますが、合っていますか？

```

**結果：最初の認識ズレを防げる → 手戻りがゼロになる**

---

### 課題6：修正のたびに全体が再出力されると、どこが変わったかわからない

**例えば...**

- ユーザー：「カテゴリ2だけ修正して」
- AI：「わかりました！全10カテゴリを再出力します」
- → **どこが変わったかわからず、確認に時間がかかる**

**解決策**
→ **指定カテゴリのみ修正版を出力**

```
【修正版：カテゴリ2】
[更新内容]

理由：●●を反映しました。

他に修正はありますか？

```

**結果：差分が明確で、確認が楽になる**

---

### 課題7：リスクが見えず、プロジェクトが失敗する

**例えば...**

- 完璧な事業計画を作ったつもりが、実行段階で想定外の問題が発生
- 「こんなリスクがあるなんて知らなかった...」
- → **ポジティブな提案だけだと、落とし穴に気づかない**

**解決策**
→ **完成時に「リスク別記」を必ず付記**

```
## リスク別記

### うまくいかない要因
- 税務・法務・運用・コスト・人的・心理・社会的リスク
- プロジェクト完了時の振り返りで想定される失敗要因

### 対策
- リスク軽減策を具体的に提示

```

**結果：プリモーテム分析の観点で、事前にリスクを把握できる**

---

## このプロンプトの全体フロー（まとめ）

```
1. 領域判定 → ユーザー確認
   「新規事業ですね？」

2. カテゴリ提案（表形式）
   「市場分析・競合分析・BMC...が必要です」

3. リサーチ提案 → ユーザー許可
   「市場データはWeb検索で取得できます。やりますか？」

4. 一括たたき台生成
   全カテゴリを一気に出力
   - リサーチした情報 → 自動で埋める
   - ユーザー固有情報 → [要ヒアリング] で残す

5. 逐次修正
   「カテゴリ2だけ修正」→ カテゴリ2のみ更新

6. ファイル出力
   各カテゴリを独立ファイルで保存
   + リスク別記を付記

```

---

## なぜこの設計にしたのか？（設計思想）

### 思想1：ユーザーの納得強度を最大化する

- **推論は積極的に行うが、決定前に必ず確認を取る**
- 「なぜそうするのか」を理解した上で進められる
- → ブラックボックス化を防ぎ、信頼を得る

### 思想2：効率と品質を両立する

- **たたき台一括生成で時間を短縮**
- **リサーチとヒアリングを適切に分担**
- → 時間をかけずに、高品質な成果物を作る

### 思想3：逐次改善（Iterative Refinement）

- **完璧を目指さず、対話で磨く**
- **部分修正で素早くフィードバックループを回す**
- → 一発完璧を目指さない、現実的なアプローチ

### 思想4：構造化とリスク意識

- **論理的で読みやすい構造を保つ**
- **リスク別記で、失敗要因を事前に可視化**
- → ポジティブな提案だけでなく、落とし穴も見せる
